Daily Workflow

Eight steps from starting work to clean-up. Every feature, every bug fix, every time.

01
Start from fresh dev
Always begin by pulling the latest development branch. Never work from stale code.
git checkout dev && git pull
02
Create a feature branch
Every change gets its own branch. Name it clearly — this becomes the PR title context.
git checkout -b feature/add-audio-extraction
03
Build the feature
Write the code and write the tests in the same branch. Tests are not a follow-up task — they ship with the code, always.
Never create a separate "add tests later" PR. Code + tests = one unit of work.
04
Verify locally
Run the full test suite. Pre-commit hooks check formatting, linting, and secrets automatically on commit.
pytest -v
05
Push and create PR
Push the branch. The PR template auto-fills with what/why/how-to-test/checklist. Fill it out.
git push -u origin feature/add-audio-extraction
06
CI runs automatically
GitHub Actions kicks off three jobs: lint, test, and security scan. All must pass before the PR can merge.
CI fails if coverage drops below 80% or if formatting/linting issues exist.
07
Merge on GitHub
Never merge locally. Always through a PR on GitHub. Update CHANGELOG.md before merging.
08
Clean up
Switch to main, pull latest, delete merged branches — locally and remotely.
git-done

Branching Strategy

Three long-lived branches. Short-lived feature branches. PRs flow upward.

main Production-ready code. Deploys to production. Always stable.
staging Pre-production testing. Validates before promotion to main.
dev Active development. Feature branches merge here first.
feature/* Short-lived. One per task. Created from dev, merged back to dev.
PR Flow Direction
feature/* → dev → staging → main
Never develop directly on main, staging, or dev. Always branch.

CI Pipeline

Three jobs run on every push and every PR. All must pass before merge.

1. Lint
  • black — formatting check
  • ruff — linting + auto-fix
  • Fails if code isn't formatted
2. Test
  • pytest with coverage
  • 80% coverage threshold
  • HTML report + JUnit XML
  • Coverage summary in PR
3. Security
  • detect-secrets — leaked keys
  • pip-audit — dependency CVEs
  • bandit — Python code scan

CI Outputs

Test Results
Pass/fail summary on PR checks tab via JUnit XML. Download reports/test-report.html for full detail.
Coverage Map
Line-by-line coverage in reports/coverage/index.html. Summary printed in GitHub Actions job summary.
Security Report
Python code scan in reports/bandit-report.json. Dependency vulnerabilities flagged inline.
All artifacts retained for 30 days. Dependabot opens weekly PRs for dependency updates.

Testing Requirements

Tests ship with code. Always. No exceptions.

+
New Feature
Unit tests + integration tests in the same PR as the feature code.
!
Bug Fix
Write a test that reproduces the bug first. Then fix the code. Same PR.
~
Changed Behavior
Update existing tests to match the new behavior. Same PR.
-
Removed Feature
Remove the feature's tests. Don't leave dead tests behind. Same PR.
The Testing Pyramid
E2E
Few — full browser flows
Integration
Moderate — API-level tests
Unit Tests
Many — pure logic, fast
Quality over percentage. Prioritize error paths, boundary values, state transitions, and integration points. Don't waste time testing trivial getters or framework internals.
Tests are permanent. They are regression tests. A future change that breaks an old test is the system working correctly.

Pre-commit Hooks

Nine hooks run automatically before every commit. Catches problems before they reach CI.

HookWhat It Does
trailing-whitespaceRemoves trailing whitespace from all files
end-of-file-fixerEnsures every file ends with a newline
check-yamlValidates YAML syntax (catches CI config errors)
check-added-large-filesBlocks files over 500KB from being committed
check-merge-conflictCatches unresolved merge conflict markers
detect-private-keyPrevents accidentally committing private keys
blackAuto-formats Python code to consistent style
ruffLints Python code and auto-fixes common issues
detect-secretsScans for API keys, passwords, and credentials

Build Report

Every PR gets a consolidated build report posted as a sticky comment. Updates on each push. Full history preserved.

What the Build Report Shows
Test Results

Skipped 1
Errors 0
Total 50
Coverage
Overall 87.3%
Threshold 80%
Files below 80% are listed individually with their coverage percentage.
Sticky PR Comment
Report is posted as a comment directly on the PR. It updates on each push — you always see the latest results without leaving the PR page.
Step Summary
Same report is written to GitHub Actions step summary. Every build is preserved in the Actions history for long-term reference.
Failure Details
Failed tests are listed with their error messages directly in the report. No need to dig through CI logs to find what broke.
Downloadable Artifacts
Full HTML test report, interactive coverage map, JUnit XML, and security scan results available as downloadable artifacts for 30 days.

Review Gates

Every PR requires two independent reviews before merge. Code quality and test quality are reviewed separately by different roles.

Gate 1
Dev Lead Review
  • ● Code follows project conventions and style
  • ● Architecture and design are sound
  • ● No unnecessary complexity or over-engineering
  • ● Error handling is appropriate
  • ● Logging follows the standard
  • ● No security concerns
Gate 2
QA Lead Review
  • ● Tests cover the right scenarios (happy path, errors, edges)
  • ● Tests are meaningful — not testing trivial code
  • ● Test names clearly describe what they verify
  • ● Coverage has not dropped below 80%
  • ● No flaky or environment-dependent tests
  • ● Regression tests updated if behavior changed
Review Flow
PR Created → CI Passes → Dev Lead Review → QA Lead Review → Merge
Both reviews must be completed and signed off in the PR template before merge. Neither is optional.

When Notifications Fire

Push to dev CI result + commit link + author
Push to staging CI result + commit link + author
Push to main Deploy notice + commit link + author
How to Activate
1
Create a Slack incoming webhook (Apps → Incoming Webhooks → Add to channel)
2
Add webhook URL as a GitHub Actions secret: SLACK_WEBHOOK_URL
3
Done — notifications start automatically on the next CI run
The notification job is pre-wired in CI from setup.sh. It skips gracefully if the secret isn't set — nothing breaks.

Definition of Done

A feature is complete when every box is checked. Not before.

  • ✓Code is written and follows project conventions (black, ruff clean)
  • ✓Unit tests + integration tests included in the same PR
  • ✓All tests pass locally (pytest -v)
  • ✓Coverage has not dropped below 80%
  • ✓PR description follows the template (what, why, how to test, checklist)
  • ✓CI passes — lint + test + security scan
  • ✓Build report shows no regressions
  • ✓Dev Lead has reviewed and approved (code quality)
  • ✓QA Lead has reviewed and approved (test quality)
  • ✓CHANGELOG.md is updated
  • ✓PR is reviewed and merged via GitHub
  • ✓Linked requirement or issue is updated or closed
